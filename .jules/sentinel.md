## 2025-02-14 - Fix Missing `userId` in Patient Creation for Row-Level Access
 **Vulnerability:** Patient documents were being created without a `userId` field. While `firestore.rules` permitted read/write for any authenticated user (`request.auth != null`), strict LGPD compliance necessitates row-level validation. Newly created records wouldn't have the `userId` attribute, breaking future strict access control implementations and potentially allowing unintended broad access.
 **Learning:** In Firebase v9+, accessing `auth.currentUser.uid` immediately without awaiting auth state initialization can lead to race conditions where the user appears unauthenticated (null). Relying on client-side authentication without saving the user context alongside the data bypasses the ability to enforce strict row-level authorization in Firestore rules.
 **Prevention:** Always include `userId` in newly created critical documents (like `patients`). Always use `await auth.authStateReady()` before accessing `auth.currentUser` to ensure the authentication state is fully resolved, and explicitly handle the case where `auth.currentUser` might be null to prevent unhandled exceptions.
