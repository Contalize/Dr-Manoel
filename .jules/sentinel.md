## 2026-03-17 - Hardcoded Firebase Config API Key\n**Vulnerability:** Found a hardcoded `apiKey` in `src/firebase/config.ts`. Even though Firebase keys are technically public identifiers for the web SDK, explicitly storing them hardcoded is generally discouraged and flags security scanners. However, according to the memory guidelines for this project, securing them via environment variables provides no real security benefit. Therefore, we should skip fixing the hardcoded API key and find another issue.\n**Learning:** In this project, Firebase keys are considered public and 'fixing' them is security theater.\n**Prevention:** Document this and focus on real vulnerabilities.
## 2026-03-17 - Missing userId in Document Creation\n**Vulnerability:** Found that newly created documents in Firestore (e.g., in `src/components/prescriptions/NewPrescriptionDialog.tsx`, `src/app/settings/page.tsx`, `src/app/anamnesis/page.tsx`, `src/app/patients/page.tsx`, `src/app/patients/[id]/page.tsx`) are missing the `userId` field. This contradicts the memory context which states: 'All newly created documents across collections must include a `userId` field matching the authenticated user's UID... to ensure read/write access.' This currently forces the Firestore rules to be overly permissive.\n**Learning:** Technical debt in data models can lead to insecure, permissive database rules.\n**Prevention:** Ensure all database writes from the client explicitly include the authenticated user's ID to allow strict row-level security rules.
